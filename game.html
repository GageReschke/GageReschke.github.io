Overview

Coat-of-Arms is a turn-based 2D strategy game I'm building from scratch in C++ using Raylib for graphics — no game engine. Inspired by games like Fire Emblem, Polytopia, and Civilization, the goal is to create a deeply strategic experience where complexity lives within each unit rather than in sheer unit count.

Players develop land, harvest resources, train armies, and use elemental magic spirits to dominate their opponent's territory. With only 5 unit types but 10+ stats per unit and a layered magic system, every decision carries weight.

Technical Highlights

  • Built entirely in C++ with no game engine — all game logic, rendering, input handling, and state management written from scratch
  • Implemented event-based architecture patterns to manage game state transitions cleanly across turns, combat phases, and building actions
  • Developed a shortest-path movement system with tile cost weighting for different terrain types (grassland, water, mountains)
  • Built a battle forecast system that calculates and previews combat outcomes before a player commits to an attack
  • Designed a movement snapshot system to allow previewing unit range without committing movement
  • Integrated Raylib for isometric tile rendering, sprite management, and UI overlay elements

Game Systems

The game is structured around four interlocking systems:

  • Construction — build structures that train troops, collect resources, or block enemies. Buildings like watchtowers and barracks shape your strategy
  • Resources — food, wood, and metal are harvested by extractors and spent on units and buildings, creating an economy to manage
  • Training — units take time to deploy from cities, sped up by barracks, adding a planning dimension to army building
  • Magic Spirits — four elemental spirits (Flame, Shadow, Martial, Gale) each grant unique abilities like invisibility, fire terrain, unit buffs, and positional manipulation

Martial and Gale spirit cards — each spirit provides unit-class-specific abilities

What I'm Learning

  • Applying structured software architecture — event-driven design keeps game logic modular and testable, the same discipline used in hardware development
  • Writing and debugging complex systems without the safety net of a game engine forces deep understanding of memory, rendering, and state management
  • Balancing game design with technical constraints — features like the spirit system require careful data modeling to stay extensible
  • UX design for complex systems — designing controller/input pathways so multi-step actions feel intuitive

Why this matters as an engineer

Building a game from scratch in C++ without a framework is the same muscle as engineering hardware from first principles — you have to understand every layer, debug systematically, and iterate toward a working system. The test-fix-iterate loop here is identical to bench testing a circuit or validating a mechanical assembly.
